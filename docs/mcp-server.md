# MCP-сервер kusec

MCP-сервер (Model Context Protocol, stdio transport) поверх HTTP API kusec — для AI-агентов
(Claude Code и др.). Агент видит структуру app/secret/configmap/item и управляет ею, но
**значения секретов не видит никогда** — ни существующие, ни создаваемые.

- Бинарник: `cmd/mcp` (`make build-mcp` → `cmd/build/mcp`).
- Код: `internal/mcpserver` (сервер + value-механика), `internal/mcpserver/client` (HTTP-клиент API).

## Модель безопасности

1. **Чтение**: у item-ов секретов поле `value` вырезается; вместо него — `value_chars`
   (длина в символах), `value_bytes` и `value_sha256` (усечённый sha256 — для сравнения
   значений между собой). Значения item-ов **конфигмапов** не секретны и видны полностью.
2. **Запись значений** — только декларативно через `value_source`; значение существует
   лишь в памяти MCP-сервера и уходит напрямую в API:
   - `generate` — случайное значение через crypto/rand (`format`: `alnum` | `ascii` |
     `digits` | `hex` | `base64url` | `uuid`; `length`, по умолчанию 32). При указании
     `name` сохраняется в реестре сессии для переиспользования.
   - `reuse` — взять значение из реестра сессии по `name` (тот же токен для нескольких
     item-ов одного app). Реестр живёт в памяти процесса и изолирован по app.
   - `copy_item` — скопировать значение существующего item-а по `item_id` (можно из
     любого app). С `name` — также зарегистрировать для reuse.
   - `literal` — явное значение от агента (для несекретного: хосты, порты, url).
3. **Scope записи**: все create/update выполняются только в «текущем app», который агент
   выбирает через `use_app` (создание app через `create_app` делает его текущим).
   Чтение — без ограничений. Delete-инструментов нет.
4. **Ошибки** API фильтруются: наружу уходят только `code`/`message`/`fields`, из текста
   вычищаются все значения секретов, прошедшие через процесс (`[REDACTED]`);
   неразобранные тела ответов не показываются вовсе.

Ограничения модели: усечённый хэш позволяет подтвердить совпадение значения со словарным
перебором — для сгенерированных (высокоэнтропийных) значений это безопасно, но не
защищает слабые пароли, заведённые вручную. Точная длина значения также раскрывается —
это осознанный выбор.

## Инструменты (22)

| Группа | Инструменты |
|---|---|
| Чтение | `list_app`, `get_app`, `list_secret`, `get_secret`, `list_configmap`, `get_configmap`, `list_item`, `get_item` (маскированные), `list_config_item`, `get_config_item` |
| Контекст | `use_app`, `current_app` (текущий app + имена значений в реестре) |
| Запись | `create_app`, `update_app`, `create_secret`, `update_secret`, `create_configmap`, `update_configmap`, `create_item`, `update_item`, `create_config_item`, `update_config_item` |

Пагинация zero-based (`page` с 0), `page_size` по умолчанию 100.

## Конфигурация (env)

| Переменная | Описание |
|---|---|
| `KUSEC_MCP_API_URL` | Базовый URL HTTP API, включая `/api` (например `https://kusec.example.com/api`). Обязательна. |
| `KUSEC_MCP_API_KEY` | **Рекомендуемый способ**: статичный API-ключ (`ksk_…`), не протухает, отзывается в любой момент. Как выпустить — ниже. |
| `KUSEC_MCP_REFRESH_TOKEN` | Долгоживущий refresh-токен (access-токен обновляется автоматически, ротация refresh-токена поддерживается). |
| `KUSEC_MCP_USERNAME` / `KUSEC_MCP_PASSWORD` | Логин/пароль сервисного аккаунта **без 2FA**. |
| `KUSEC_MCP_INSECURE_SKIP_VERIFY` | `true` — не проверять TLS-сертификат (только для локалки). |

Права в kusec действуют как обычно: MCP-сервер работает от имени пользователя, которым
авторизован (набор доступных app, admin-флаги и т.д.).

## API-ключи (`/api-key`)

Долгоживущие учётные данные для машинных клиентов. В БД хранится только sha256-хэш;
значение ключа возвращается **один раз** при создании. Ключ наследует права владельца
(usr): его `app_ids` и `is_admin`. 2FA на api-key-аутентификацию не распространяется,
поэтому ключи стоит выпускать на сервисного пользователя с минимальным набором app.

```bash
# выпустить ключ (Bearer — обычный JWT после логина)
curl -X POST https://kusec.example.com/api/api-key \
  -H "Authorization: Bearer $JWT" \
  -d '{"name": "mcp local mdb"}'
# → {"id": "...", "key": "ksk_<hex>"}  — key больше нигде не получить

# список своих ключей (админ видит все, есть фильтр usr_id)
curl https://kusec.example.com/api/api-key?list_params.page_size=100 -H "..."

# отключить / удалить
curl -X PUT    .../api/api-key/{id} -d '{"active": false}' -H "..."
curl -X DELETE .../api/api-key/{id} -H "..."
```

Не-админ управляет только своими ключами; админ может выпускать ключи другим
пользователям (`usr_id` в Create). `last_used_at` обновляется при использовании
(не чаще раза в минуту). Отзыв: `active=false` или удаление — действует сразу.

## Аутентификация агентов к самому MCP

MCP-сервер работает по **stdio**: его запускает MCP-клиент (Claude Code и т.п.) как
локальный дочерний процесс. Отдельной аутентификации агента здесь нет и не требуется —
доступ получает тот, кто может запустить процесс с заданными env. Граница доверия:

- **Агент не видит ключ через инструменты MCP** — ключ живёт в env процесса и в ответы
  инструментов не попадает.
- **Не кладите ключ открытым текстом в `.mcp.json` внутри репозитория** — агент может
  прочитать этот файл обычным чтением файлов. Используйте подстановку из окружения
  (`"KUSEC_MCP_API_KEY": "${KUSEC_MCP_API_KEY}"`) или пользовательский конфиг вне репо.
- Локальный агент с полным доступом к шеллу в принципе может добраться до окружения
  пользователя — это общее ограничение локальных MCP: с добытым ключом можно пойти в
  API kusec напрямую, мимо маскирования. То есть сейчас маскирование — защита от
  случайного попадания секретов в контекст/логи LLM, а не жёсткая граница.

### План: двухключевая схема (не реализовано)

Согласованная схема жёсткой границы — MCP-сервер разворачивается рядом с kusec:

1. У api-ключа появляется флаг `mcp_only`: такой ключ kusec принимает **только в паре**
   с ключом MCP-сервера. Агентские ключи выпускаются с `mcp_only=true` — сами по себе
   они бесполезны, даже если агент достанет их из окружения.
2. Серверный ключ MCP — в env kusec (`MCP_PROXY_KEYS`, список для ротации) и в конфиге
   MCP-сервера (`KUSEC_MCP_PROXY_KEY`); на машины агентов он не попадает. Передаётся
   заголовком `X-Mcp-Proxy-Key` (нужен проброс заголовка в grpc-gateway).
3. MCP-сервер получает HTTP-режим (streamable HTTP): пользовательский ключ приходит
   сквозным в `Authorization` от MCP-клиента (`"headers": {"Authorization": "Bearer
   ${KUSEC_MCP_API_KEY}"}` в `.mcp.json`), серверный добавляется к каждому запросу в
   kusec. Реестр `reuse`-значений становится per-session. Stdio-режим остаётся для
   локального запуска с обычным (не-`mcp_only`) ключом.

Итог: личность/права — от пользовательского ключа, доверие к прослойке с маскированием —
от серверного; экфильтрация значений возможна только при компрометации самого сервера.

## Подключение к Claude Code

`.mcp.json` проекта (или `claude mcp add`):

```json
{
  "mcpServers": {
    "kusec": {
      "command": "/path/to/kusec/cmd/build/mcp",
      "env": {
        "KUSEC_MCP_API_URL": "https://kusec.example.com/api",
        "KUSEC_MCP_API_KEY": "${KUSEC_MCP_API_KEY}"
      }
    }
  }
}
```

Типовой сценарий агента:

```
use_app {"app": "billing"}
create_secret {"slug_name": "db", "description": "Postgres"}
create_item {"secret_id": "<id>", "key": "POSTGRES_PASSWORD",
             "value_source": {"kind": "generate", "name": "db_password", "length": 32}}
create_item {"secret_id": "<id2>", "key": "DATABASE_URL_PASSWORD",
             "value_source": {"kind": "reuse", "name": "db_password"}}
```
